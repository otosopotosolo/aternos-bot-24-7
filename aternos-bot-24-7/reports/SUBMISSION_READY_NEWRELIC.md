## Title
New Relic API CORS Misconfiguration - Arbitrary Origin Reflected with Credentials

## Severity
HIGH

## Summary
The New Relic API at https://api.newrelic.com/v2/applications.json allows arbitrary cross-origin requests from any origin with credentials enabled. The server reflects the Origin header value back in the Access-Control-Allow-Origin response header.

## Steps to Reproduce

1. Send a preflight OPTIONS request to https://api.newrelic.com/v2/applications.json:
```bash
curl -I -X OPTIONS 'https://api.newrelic.com/v2/applications.json' -H 'Origin: https://evil-test.com' -H 'Access-Control-Request-Method: GET'
```

2. Observe the response headers:
```
HTTP/2 204
access-control-allow-origin: https://evil-test.com
access-control-allow-methods: DELETE, GET, HEAD, OPTIONS, PATCH, POST, PUT
access-control-allow-credentials: true
```

## Impact
An attacker can craft a malicious webpage that makes XMLHttpRequest/fetch calls to the New Relic API on behalf of logged-in users, potentially stealing:
- Application monitoring data
- Performance metrics
- Infrastructure information
- Account configurations

## Remediation
Implement strict CORS policy by:
1. Using a whitelist of allowed origins (*.newrelic.com)
2. Not reflecting arbitrary Origin headers
3. Implementing proper origin validation
