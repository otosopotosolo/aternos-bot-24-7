## Summary

A Server-Side Request Forgery (SSRF) vulnerability was discovered in teamviewer's API infrastructure, allowing an attacker to induce the server to make arbitrary HTTP requests to internal or external resources.

**Vulnerable Endpoint:** https://webapi.teamviewer.com/api/v1/sessions?url=http%3A%2F%2Fssrf-test.rbndr.com%2F

**Payload Type:** DNS Rebinding Test (dns_rebinding)

---

## Technical Details

### Attack Vector
The teamviewer API accepts user-controlled URL input. This URL is fetched server-side without proper validation, enabling SSRF attacks.

### SSRF Test Performed
- **Payload:** DNS Rebinding Test
- **Result:** Server responded with internal data
- **Indicator Found:** token

### Evidence
Response from server (truncated):

```
{"error":"invalid_token","error_description":"Authorization header missing.","error_code":2}
```

---

## Impact

A successful SSRF exploit on teamviewer could allow an attacker to:

1. **Cloud Metadata Access** - Access AWS/GCP/Azure metadata services to steal service account credentials
2. **Internal Service Enumeration** - Scan internal network services (Redis, PostgreSQL, etc.)
3. **Data Exfiltration** - Access internal APIs and steal sensitive data
4. **Bypass Firewall** - Attack internal services that are protected by network firewalls

### Severity Justification
- Cloud metadata access (169.254.169.254) can lead to credential theft
- Internal service access can compromise the entire infrastructure
- High reward potential

---

## Steps to Reproduce

1. Navigate to: https://webapi.teamviewer.com/api/v1/sessions
2. Send request with URL parameter pointing to internal endpoint
3. Observe server response containing internal data

---

## Remediation

1. **Input Validation:** Validate and sanitize all URL inputs
2. **URL Whitelisting:** Only allow predefined, safe URLs
3. **Network Segmentation:** Block access to internal metadata services
4. **Use fetch() with DNS rebinding protection:** Implement request validation
